1. What is amqp?

AMQP (Advanced Message Queuing Protocol) is an open standard networking protocol designed for passing business messages between different applications or organizations. In event driven system, it acts as the universal language that allows publisher and subscriber to communicate through a central message broker (in this case, RabbitMQ). By using this protocol, the programs remain decoupled. The publisher can send data (events) to the broker without needing to know who the subscriber is or if the subscriber is even currently running. This ensures that the system can handle traffic reliably, as AMQP manages the logistics of routing and storing messages in a queue until they can be processed.

2. What does it mean? guest:guest@localhost:5672 , what is the first guest, and what is the second guest, and what is localhost:5672 is for?

The connection string guest:guest@localhost:5672 is essentially the home address and key that the subscriber uses to find and log into the RabbitMQ broker. The first guest in the string represents the username, while the second guest represents the password. These are the default administrative credentials provided by RabbitMQ for development environments. Using these default settings simplifies the setup process while we are building the initial code, though in a production environment, we would eventually replace these with more secure, unique credentials.
<br>The latter portion of the string (localhost:5672) defines the physical location of the broker on the network. Localhost specifies that the RabbitMQ instance is running on the same computer as the code, inside the Docker container we started earlier. The number 5672 is the specific TCP port that RabbitMQ listens on for incoming AMQP traffic. Together, these components ensure that the Rust program knows exactly which door to knock on and what password to provide to begin listening for new events.

The screenshot illustrates the slow subscriber simulation running in a local environment, where the discrepancy between the production and consumption speeds is clearly visible. The "Message rates" chart shows a sharp, early spike in the Publishing rate, indicating that the publisher sent a burst of messages almost instantaneously. In contrast, the Deliver rate remains flat and low at approximately 1.0 message per second. This occurs because the subscriber is intentionally throttled with a delay, such as a one-second sleep, forcing it to process only one message at a time while the publisher has already finished its entire task and exited.

The total number of queued messages in this simulation reaches a peak of approximately 25, as seen in the red "Total" line of the top chart. This number is as such because it represents the exact volume of messages the publisher broadcast that the subscriber has not yet processed. Since the publisher is much faster than the slow subscriber, the broker must store these 25 messages in a Ready state within the queue. This is a perfect example of Load Leveling, where RabbitMQ acts as a buffer to protect the subscriber from being overwhelmed, holding the messages safely in memory and delivering them one by one until the queue eventually returns to zero.

The terminal windows demonstrate the Competing Consumers pattern, which is a fundamental strategy for scaling distributed systems horizontally. By running three separate instances of the subscriber, the workload is automatically shared across all active consumers. RabbitMQ utilizes a Round-Robin distribution method, which explains why the messages are interleaved across the different terminals rather than appearing in a single window. The dashboard provides the administrative confirmation of this setup, showing that the system has successfully registered three connections and three consumers. Because the consumers are working in parallel, the queue clears much faster than a single-consumer setup, as represented by the shorter, sharper decline in the "Queued messages" chart compared to previous "slow" tests.

Examining the source code reveals several critical areas for improvement, starting with the naming convention used for the artificial delay. On line 17 of the subscriber's 'main.rs', the variable is named 'ten_millis', yet the value is set to 1000 (one second), which creates "logic debt" and can cause significant confusion during future debugging or performance tuning. A professional-grade refinement would be the implementation of Quality of Service (QoS) by setting a 'prefetch_count' of 1. This prevents the broker from blindly pushing messages to a busy worker and ensures that the workload is distributed based on actual availability rather than just a simple rotation. Furthermore, the hardcoded strings for identifiers and connection URLs should be extracted into an environment file or a central configuration module to improve security, portability, and maintainability as the project moves from local development to the cloud.
