## Questions

1. What is amqp?

AMQP (Advanced Message Queuing Protocol) is an open standard networking protocol designed for passing business messages between different applications or organizations. In your event driven system, it acts as the universal language that allows your publisher and subscriber to communicate through a central message broker (in this case, RabbitMQ). By using this protocol, the programs remain decoupled. The publisher can send data (events) to the broker without needing to know who the subscriber is or if the subscriber is even currently running. This ensures that your system can handle traffic reliably, as AMQP manages the logistics of routing and storing messages in a queue until they can be processed.

2. What does it mean? guest:guest@localhost:5672 , what is the first guest, and what is the second guest, and what is localhost:5672 is for?

The connection string guest:guest@localhost:5672 is essentially the home address and key that your subscriber uses to find and log into the RabbitMQ broker. The first guest in the string represents the username, while the second guest represents the password. These are the default administrative credentials provided by RabbitMQ for development environments. Using these default settings simplifies the setup process while you are building the initial code, though in a production environment, you would eventually replace these with more secure, unique credentials.
<br>The latter portion of the string (localhost:5672) defines the physical location of the broker on your network. Localhost specifies that the RabbitMQ instance is running on the same computer as your code, inside the Docker container you started earlier. The number 5672 is the specific TCP port that RabbitMQ listens on for incoming AMQP traffic. Together, these components ensure that your Rust program knows exactly which door to knock on and what password to provide to begin listening for new events.

By running three subscriber instances, the event processing load is distributed across multiple independent programs instead of relying on a single processor. Each subscriber consumes a different message from the RabbitMQ queue, which is why the terminal logs show the workload being split among the different windows. This parallel processing explains why the spike in the "Queued messages" chart on the RabbitMQ dashboard drops back to zero much faster than when only one subscriber was active. Even with the simulated one-second delay per message, the total throughput of the system is effectively tripled, preventing the queue from becoming a bottleneck. This architectural pattern allows the application to remain responsive and avoid crashes during high demand periods, such as the SIAK War scenario. The message broker facilitates this scaling seamlessly because the publisher never needs to know how many subscribers are currently processing the data.

One significant improvement would be moving the UserCreatedEventMessage struct into a separate, shared library crate to prevent code duplication between publisher and subscriber projects. Currently, both projects define the exact same structure manually. If I were to change a field in the publisher but forget to update the subscriber, it would lead to desynchronization and runtime errors. In the subscriber's main function, replacing the loop {} with a more efficient signal-waiting mechanism would reduce CPU usage compared to a busy-wait loop. I should also consider replacing the frequent use of '.unwrap()' with proper error handling (like match or if let) to prevent the entire program from panicking if the connection to RabbitMQ is temporarily lost. Additionally, implementing a dead-letter queue logic for the subscriber would ensure that messages that fail to process are safely stored for later review rather than simply being lost. Finally, updating the libraries to their latest versions would offer better performance and access to modern Rust features, though it would require minor adjustments to the trait implementations.
